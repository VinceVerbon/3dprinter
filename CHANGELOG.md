# Changelog

All notable changes to **3dprinter** are documented here. Format: [Keep a Changelog](https://keepachangelog.com/). Version source of truth: `VERSION` at repo root. SemVer; pre-1.0 treat MINOR like post-1.0 MAJOR.

## [Unreleased]

### Fixed
- **Tauri installer now builds on Windows (first shipped artifact).** `npm run app:build` failed because the script invoked the CLI as `app/node_modules/.bin/tauri` — a forward-slash path with no `.cmd` extension that Windows `cmd.exe` can't resolve (`'app' is not recognized`). Changed it to bare `tauri` (npm puts `node_modules/.bin` on `PATH`, so it resolves cross-platform even after `cd ..`); same fix for `app:dev`. Also synced the stale version (`0.4.0` → `0.8.0`) in `app/package.json`, `src-tauri/tauri.conf.json`, and `src-tauri/Cargo.toml` so the bundle is named correctly. Produces `Haspel_0.8.0_x64-setup.exe` (per-user NSIS) + `Haspel_0.8.0_x64_en-US.msi` (per-machine); smoke-tested the built `haspel.exe` (sidecar binds 127.0.0.1:5174, reads `%APPDATA%\Haspel\data`).

## [0.8.0] — 2026-05-25

### Added
- **Brand logos: "Fetch all missing" bulk action.** The Brand logos page now has a one-click button (in the header, labelled with the outstanding count) that auto-fetches logos for **every** brand that has no logo yet or whose previous fetch came back empty (`missing`). It runs up to 4 in parallel, shows live progress (`Fetching 3/12… (2 found)`), reuses the per-row spinner, and writes once at the end (single atomic save, no write races). Brands explicitly set to *Text only* are left alone (deliberate choice). Verified at runtime: 6 seeded brands → 5 resolved via favicon/Clearbit, 1 bogus brand correctly recorded `missing`, count button updates to the remaining total.

### Changed
- **Editing a filament now scrolls the edit form into view.** The add/edit form renders at the top of the Filaments page; clicking *Edit* on a card lower down previously left the viewport on the card, so it wasn't obvious you'd entered edit mode. Selecting *Edit* (or *Add filament*) now smooth-scrolls to the top of the form. Verified at runtime: from a scrolled-down list, clicking Edit brings the form to the top of the viewport.

### Fixed
- **Navigation silently died after a PWA update (dead nav links / a dead "Add a printer" button).** Routes are code-split, so each page loads as a lazy `import()`. With the PWA's `registerType: 'autoUpdate'`, when a new build's service worker activates it purges the previous build's precached chunks; a tab still running the old bundle then requests a route chunk whose hashed filename no longer exists anywhere, the dynamic import rejects, and vue-router aborts the navigation **without any error or visible feedback**. The symptom was a nav link or the first-run "Add a printer" prompt appearing to do nothing (you stayed on Filaments). Added a `router.onError` self-heal in `app/src/router/index.ts`: a chunk-load failure now hard-reloads to the intended route (served by the now-active service worker, i.e. a consistent build), with a 10 s guard against reload loops. Proven at runtime by forcing the exact failure (service worker blocked, `/printers` chunk removed) and confirming the app recovers onto the Printers screen; verified no spurious reloads occur on a healthy build.

## [0.7.0] — 2026-05-24

### Changed
- **Brand-store shopping is now AI-inferred per make+model — no more store scraping.** The v0.6.x approach (scrape the Bambu store HTML + fetch each product page for prices) was unreliable: it rate-limited the store, prices aren't server-rendered, and only the P2S had a real per-model collection (P1S/X1/A1 collections were near-empty). Replaced with a single AI-inference call: given the owned printer's **brand + model**, the helper asks the configured model for the spare parts / consumables / add-ons that brand's store sells **tailored to that exact model**. It covers each model's multi-material unit (Bambu AMS / AMS 2 Pro, Anycubic ACE / ACE Pro, Prusa MMU3, Mosaic Palette, Creality CFS — `ams` category, omitted when the model has none) and brand-specialty add-ons (spool holders, filament dryers/warmers, enclosure kits, aux cooling, camera/LED — `accessory` category). **No prices** (the store doesn't expose them to a server fetch; opted out). Crucially the inference runs **tools-disabled (`--tools ""`)** so Claude does a single-turn knowledge generation instead of autonomously web-browsing — browsing took 5+ minutes and timed out; the no-tools call is ~60–130 s and reliable. Cached per `brand|model` for 24 h. The Shopping selector is now per owned **printer (make+model)**; `store-lists.json`, the staleness prompt's notification key, and the catalog picker are keyed by make+model. Verified end-to-end: **P2S** → AMS 2 Pro parts + Bambu add-ons; **Anycubic Kobra S1 Combo** → ACE Pro parts + dryer/enclosure; **Prusa MK4S** → no AMS, Prusa-specific parts; UI dropdown shows "Bambu Lab P2S" and the list renders.

## [0.6.1] — 2026-05-24

### Fixed
- **Brand-store "Update store" failed with "fetch failed: not found."** The helper registered the fetch route as `/api/fetch-store`, but the dev/preview proxy strips the `/api` prefix before forwarding (every other helper route omits it), so the real proxied request `/fetch-store` matched no route → 404. Renamed the route to `/fetch-store`. The v0.6.0 brand-store fetch was non-functional through the UI; verified fixed end-to-end (proxy returns 72 items; the "Add from store list" picker renders them with prices).
- **A corrupt `store-lists.json` no longer blanks the Shopping page.** `useStoreListsStore.load()` now coerces a non-array payload to `[]` instead of letting `.find`/`.map`/`.filter` throw and break the route render.

## [0.6.0] — 2026-05-24

### Added
- **On-demand brand-store shopping.** The shopping catalog is no longer a static preloaded list — it's fetched per brand, on demand, from the stores of the printers you own. `data/catalog/{replacement-parts,consumables}.json` now ship **empty** (`[]`); a new per-install `store-lists.json` holds fetched lists, each stamped with `fetched_at`. New helper endpoint `POST /api/fetch-store` fetches a brand store **from the user's own machine** (so it isn't blocked the way cloud IPs are) and returns a `StoreList`. For Bambu (a Next.js storefront) it reads the P2S / AMS 2 Pro collection pages for product handles, then pulls each product page's embedded schema.org data for the real **name + EUR price + SKU**, categorising by keyword — no AI tokens, ~2 s for ~70 items (prices appear where the store lists them; out-of-stock parts have none). Other brands are best-effort from a user-supplied store URL, with a graceful "add items manually" fallback. The **Shopping** page gains a brand-store dropdown, an editable store URL (prefilled with the brand's EU store base), an intentional **Update store** button (never auto-fetches — no wasted tokens/bandwidth), a per-store filter, and a "last updated N days ago / stale" indicator. New `StoreItem` / `StoreList` types, `useStoreListsStore`, `useStoreFetch`, and a reworked `CatalogPicker` sourcing from fetched lists.
- **Store-staleness prompt + notification management.** A store list older than **30 days** raises a prompt — **Update now? / No / Ask me in a week / Disable this notification** — backed by a general keyed notification state (`disabled` + `snoozed`) in settings and a shared `useNotifications` composable. **Settings → Notifications** lists disabled/snoozed notifications with per-item re-enable and a "reset all" action. New `StaleStorePrompt.vue` (mounted globally in `App.vue`), `NotificationState` type.

### Fixed
- **Printer catalog brand-store links now point to the EU store.** All 10 Bambu `store_url`s in `data/catalog/printers.json` pointed at the US store (`us.store.bambulab.com`); switched to `eu.store.bambulab.com` to match the app's EU/NL orientation. The X1-Carbon slug was also corrected (`bambu-lab-x1-carbon` → `x1-carbon`, which 404'd on the EU store). Each EU product URL was verified to return HTTP 200.

- **The first-run "Add a printer" button now opens the form even when you're already on the Printers page.** It relied on an `onMounted` read of `?add=1`, which doesn't re-fire when routing to the page you're already on; switched to a `watch` on the route query.

## [0.4.0] — 2026-05-24

### Added
- **Installable native app via Tauri 2.** Haspel now packages into a Windows installer (`.msi` + `.exe`) — a ~4.5 MB Rust/WebView2 shell that bundles the Vue frontend and runs the Node helper as a **sidecar**. New `src-tauri/` (Cargo crate, `tauri.conf.json`, `main.rs` that spawns the sidecar with `PROJECT_ROOT` = the app resource dir and pipes its stdio, `capabilities/default.json` granting `shell:allow-execute` to the sidecar, icons generated from `app-icon-512.png`). The read-only `data/catalog/` + `data/demo/` seed ships as Tauri **resources**; user data stays at `%APPDATA%\Haspel\data`, so an installed Haspel sees the same inventory as the dev/PWA build. Frontend gains `app/src/lib/apiBase.ts` + a runtime fetch shim in `app/src/main.ts` that rewrites `/api`+`/data` to the sidecar's `127.0.0.1:5174` **only under Tauri** (no-op in dev/web, so the same `app/dist` works in the webview and in `vite preview`); the helper's CORS now also allows the Tauri webview origins (`http://tauri.localhost` / `tauri://localhost`). Verified end-to-end: both installers built, NSIS per-user install lands `haspel.exe` + `haspel-helper.exe` + bundled `data\catalog`/`data\demo`, the installed window renders the inventory, and the bundled sidecar is observed spawning. - **End-user documentation + production doc bundle.** A full user guide now lives as Markdown source under `docs/user-guide/*.md` (13 numbered topic files: introduction, getting started, filaments, AI & providers, colours & swatches, importing orders, labels, accessories, shopping list, empty spools, filament history, settings, troubleshooting — plus a `README.md` describing the structure). A zero-dependency builder, `scripts/build-docs.mjs`, assembles them in order into a single `Haspel-User-Guide.html` + `.pdf` (with a generated title page, clickable table of contents, and a copy of the Markdown source), rewriting inter-chapter `NN-name.md` links into in-page anchors. Output lands in `app/dist/docs/` (the gitignored production dir). `scripts/build-prod.ps1` wraps the whole production build (`vite build` + doc bundle) with `-DocsOnly` / `-SkipDocs` / `-NoPdf` flags. CLAUDE.md gains a mandatory **"Production build — documentation"** ritual: review + update the docs, then build + bundle them into the production dir, on every production build.

Enabling changes: `helper/index.mjs` makes CORS `ALLOWED_ORIGIN` env-overridable; `app/vite.config.ts` makes `VITE_PORT`/`HELPER_PORT` env-driven (defaults unchanged) and adds a `preview` proxy block. Doc: `docs/dev-testing.md`.

A **History (N)** button on the Filaments page opens an archive modal where each removed filament can be **Restored** to active inventory (re-created verbatim; a fresh id is minted only if the original id somehow clashes) or **hard-deleted permanently** (per-item, with an inline two-step confirm, plus a "Clear all history" action). New `useFilamentHistoryStore` + `FilamentHistory.vue`; `ArchivedFilament = Filament & { removed_at }`. The detail modal shows the on-spool/refill counts alongside the state cards. Order import sets the split automatically: a line whose product name says "refill"/"(Bijvullen)" lands as a refill, everything else as on-spool. Legacy records are backfilled (everything counts as on-spool) so existing data starts balanced.

### Added
- **Avery Zweckform 6138 label preset + configurable label-sheet geometry.** Label sheet layout is no longer hardcoded to Decadry 105×70mm 8-up — every dimension (paper size, columns×rows grid, label W×H, margins top/bottom/left/right, horizontal/vertical gaps, corner radius) now lives in a `LabelFormat` driven by a preset dropdown in the print panel. Two presets ship: **Decadry 105×70 mm (8-up)** (current default, unchanged behaviour) and **Avery Zweckform 6138 (105×74 mm, borderless)** added because Decadry sheets adhere poorly. An "Adjust dimensions" disclosure exposes every field as a numeric input — the preset value is the placeholder, any non-matching value becomes an override, and "Reset to preset" wipes all overrides. State persists to `localStorage` under `label.format.v1`; the legacy `label.topMarginMm` value is auto-migrated as a marginTop override on the Decadry preset and the old key is removed. `SheetPositionPicker` now adapts to `cols × rows` instead of the hardcoded 2×4. `POST /api/render-labels-pdf` accepts `{ formatId, overrides }` alongside the existing fields and round-trips them through `?formatId=…&fmt=<base64url-json>` so the headless PDF matches the on-screen panel exactly; legacy `topMargin` query param still honoured.
- **Label-printing feature** (`/labels` + `/labels/logos`). Pick one or many filaments, preview a single label at actual size, then print to a Decadry 105×70mm sheet (8-up on A4). Layout follows the reference Bambu spool label: producer wordmark/logo bottom-left, large material title top-left, color swatch + variant name top-right, spec table (material / print temp / bed temp / diameter / weight / dry schedule), provider article number bottom-right. No barcode (per user — not relevant). A prominent **AMS** chip on the label front shows compatibility with the AMS 2 Pro: green check for compatible, red strikethrough X for incompatible. The compatibility rule (`app/src/lib/labels/amsCompat.ts`): Bambu-branded filaments follow `ai.p2s_compatibility.ams2pro` directly with Bambu "TPU for AMS" being the only AMS-compatible TPU; non-Bambu TPU is never AMS-safe; non-Bambu stiff/reinforced (PA, PA-CF, PC, PLA-CF) is incompatible; non-Bambu PLA/PLA+/PETG/ABS/ASA (incl. matte/silk/sparkle/etc.) is compatible. Sheet position picker lets you start at any of the 8 slots so a partially-used sheet doesn't waste blank labels. Font is Plus Jakarta Sans (Typold-bold lookalike, MIT-licensed Google Font) — loaded via @import in `style.css`. Print is driven by a global `@page { size: A4; margin: 0 }` + `@media print` block that hides app chrome and reveals only `.print-root`.
- **Brand-logos manager** at `/labels/logos`. Lists every brand currently in inventory and lets you set its logo via four mechanisms: Auto-fetch (helper `POST /fetch-logo` tries Clearbit first, then the supplier's `apple-touch-icon` / favicon variants, returning a data-URI), Upload (any PNG/SVG/JPEG up to 1 MB, stored as data-URI in `data/brand-logos.json`), Paste URL (logo loaded at print time from the URL), or Text only (renders the brand name as a bold Plus Jakarta Sans wordmark). Per-brand state stored in `data/brand-logos.json` keyed by slug; falls back to text-only wordmark when nothing is set so labels always print *something* even before logos are configured.
- **Server-side PDF download** at `POST /api/render-labels-pdf`. Helper spawns `msedge --headless --print-to-pdf` against the live Vue app with `?ids=…&startPosition=…&topMargin=…` query params, polls for Edge's deferred PDF write (Edge exits before its renderer subprocess finishes writing — typical wait ~13 s), and returns the binary. The "Download PDF" button on `/labels` calls it and offers a download — bypasses the browser print dialog entirely so the user never deals with Chrome's default margins compressing the 297mm sheet. Top-margin and start-position are user-controllable; top-margin is persisted to `localStorage` so the dialed-in Decadry value sticks.
- **Label layout polish.** Title truncation strips supplier-style cruft (`"TPU 85A / TPU 90A — TPU 85A 1 kg with spool"` → `"TPU 85A / TPU 90A"`, `"(Bijvullen)"` → `"(refill)"`), auto-shrinks font in 4 tiers (22pt → 19 → 16 → 13) so long titles still fit, and hard-caps at 2 lines via line-clamp. Spec table is now 2 columns side-by-side, distributing six rows over the 70mm height with proper visual rhythm. `justify-content: space-between` on the label flex column eliminates the dead-zone whitespace the first cut had between specs and the bottom row. Logo `<img>` has an `@error` handler that falls back to the text wordmark, so broken URLs no longer leak alt-text into the print.

### Fixed
- **Print pagination no longer emits phantom trailing blank pages.** The `<main class="p-6">` padding from `App.vue` was adding 24px below the print-root in print mode, pushing the body slightly past 297mm and triggering Chrome to allocate a second physical page that was always blank. `@media print` now strips `#app` and `<main>` padding to zero.
- **Print row alignment.** Labels are now placed via absolute positioning per slot (`top: 8.5 / 78.5 / 148.5 / 218.5 mm`) rather than CSS-grid track sizes. Chrome's print engine silently shrank fixed `mm` grid rows when the dialog's default margins reduced the printable area, collapsing rows into each other; absolute positioning is immune. `top` is configurable per Decadry product via the new top-margin input on the print panel.

### Fixed
- **Star ratings on filament/accessory cards now reflect the stored value instead of looking editable-but-discarding.** `FilamentCard.vue` and `AccessoryCard.vue` mounted `<RatingStars>` without `readonly` and bound `@update:model-value` to a no-op, so clicking the stars on a card showed hover + updated the local model but threw the value away — the "Save doesn't store my rating" report. Both cards now match `FilamentDetail.vue`: `readonly :size="14"`. The proper edit path remains the pencil/Edit button → form (`FilamentForm.vue` / `AccessoryForm.vue` v-model the rating into the submitted record).
- **`scripts/start.ps1` failed silently on machines where Windows Hyper-V/WSL had swallowed the dynamic excluded port range across 5173/5174.** The hidden npm spawn died with `EACCES: permission denied 127.0.0.1:5173` and the launcher opened Edge anyway → "connection refused." This is environmental, not a repo bug — the fix on the affected machine is a one-time elevated `net stop winnat` + `netsh int ipv4 add excludedportrange protocol=tcp startport=5173 numberofports=1` (and 5174) + `net start winnat` to claim both ports as administered exclusions before Hyper-V grabs them. Persists across reboots. Documented in `crosslog.md` #17 so the next machine that hits this can apply the same fix without re-diagnosing.
- **The helper no longer dies under an open app ("Lookup AI → Failed to fetch").** The heartbeat watchdog killed the helper after 45 s of heartbeat silence — but browsers throttle/freeze background-tab timers, so alt-tabbing away from the PWA for ~45 s killed the helper out from under the still-open window. The next AI call then hit the Vite proxy with no upstream and surfaced the opaque "Failed to fetch." In dev mode the helper now ties its lifetime to the **vite process** (`process.kill(VITE_PID, 0)` liveness probe) instead of heartbeats — vite already spawns and reaps it — so it stays alive for the whole session regardless of tab visibility. Standalone mode keeps a heartbeat watchdog but with a generous 10-min grace (was 45 s; overridable via `WATCHDOG_TIMEOUT_MS`). App also beats on `visibilitychange`/`focus` so returning to the window refreshes the standalone watchdog immediately.
- **Order import no longer creates duplicate filament entries on re-import.** `OrderImportReview` matched against in-memory inventory only when the review dialog opened; if the existing entries weren't loaded then (e.g. helper had been offline), a re-import added twin spools instead of bumping the sealed count. Apply now re-checks identity (SKU, else brand+name+variant) at apply time against current inventory and merges into the existing entry.
- **Offline edits are no longer silently lost on reload.** `loadData` only ever read the helper's file; a save made while the helper was offline went to `localStorage` and was never read back, so the file (e.g. with duplicates still present) reappeared on the next load. `loadData` now flushes any pending `offline:<file>` copy to the helper before reading, falls back to the offline copy when the helper is unreachable, and `saveData` clears the offline copy once a real save lands.
- **Import swatch resolution capped at 3 concurrent lookups.** `resolveSwatchesAsync` fired every `/lookup-swatch` at once (`Promise.all`); each spawns a heavyweight `claude` + WebFetch subprocess, so larger orders overloaded the machine and some calls timed out and silently fell back to the weaker Tier-1 hex hint — the reason a re-import (warm cache) produced visibly better swatches. Now bounded to 3 in flight with live `(n/total)` progress.

- **Helper JSON parsing is now lenient.** `runClaude` callers used to do `JSON.parse(stripFences(inner))` which would fail if claude prefaced its JSON with a sentence of prose (e.g. "Found it confirmed in the raw GitHub colors.json…  {…}"). Caught a real failure during the swatch backfill (PLA Matte refill "Appelgroen") and applies to `/lookup-filament`, `/lookup-swatch`, and `/import-order` uniformly.
- **PDF drop-zone now actually imports orders.** The frontend `OrderDropZone.vue` was wired in v0.2 but the matching helper endpoint (`POST /import-order`, Chunk F) was never implemented — drops surfaced as "Failed to fetch" in the UI because the helper's 404 fallthrough never consumed the multipart body, so Node tore the connection down mid-upload. Implemented the endpoint: streams the PDF to a temp file via `busboy` (10 MB cap → 413), shells out to `claude --print --output-format json` from `tmpdir()` (so the project's `CLAUDE.md`/crosslog can't bleed into the model's context and trip its prompt-injection guard) with `--allowedTools Read --permission-mode bypassPermissions --no-session-persistence --append-system-prompt` framing the call as the legitimate extraction backend it is, then parses the wrapper, strips fences, validates shape, and returns the contract documented in `docs/parallel-work.md` §326–356. Verified end-to-end with a synthetic receipt PDF: vendor + order_ref + date + line items (incl. SKU + variant + unit price) all populated. Temp PDF is unlinked in `finally`. New dep: `busboy@^1.6` in `helper/package.json`.
- **Save button on Add/Edit forms now actually saves.** `FilamentForm.vue` and `AccessoryForm.vue` were calling `structuredClone(form)` and `structuredClone(props.initial)`, but the form is a `reactive()` Vue Proxy whose nested objects (e.g. `swatch`, `inventory`) are also Proxies — Chromium's `structuredClone` rejects those with `DataCloneError: #<Object> could not be cloned`, so the submit handler threw before `emit('submit', …)` ever ran. The form has `@submit.prevent`, so the only user-visible symptom was "the Save button does nothing." Replaced with a JSON-roundtrip clone (`JSON.parse(JSON.stringify(v))`) since both records are pure JSON. Affects every Add/Edit path on the Filaments and Accessories pages, including the edit launched from the filament detail modal.
- **Desktop launcher (`scripts/start.ps1`)** now actually opens the app on Windows. Four bugs were stacked: (1) `Get-Command npm` returned the `npm.ps1` shim, which `Start-Process` cannot launch — now resolves `npm.cmd` first; (2) `msedge.exe` is not on PATH on this machine (Edge installs under `Program Files (x86)`), so the Edge branch was skipped — added a path-probe fallback to the standard install locations for both Edge and Chrome; (3) the script was UTF-8 without BOM, so `powershell.exe` 5.1 read the em dashes as Windows-1252 and parse-failed silently in the hidden window — re-saved with UTF-8 BOM; (4) **root cause of the user-visible failure**: `app/vite.config.ts` had `server.host: 'localhost'`, which on Windows resolves to `::1` first, leaving Vite IPv6-only — so the `127.0.0.1:5173` health probe and `msedge --app=http://127.0.0.1:5173/` both got connection-refused. Pinned Vite to `127.0.0.1` to match the helper.
