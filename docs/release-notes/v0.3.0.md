# 3dprinter v0.3.0 — Chunk E + Chunk F: the loop closes

**Date**: 2026-05-11

## Summary

v0.2 shipped the frontend for PDF order import (Chunk F) and the colour-swatch story (Chunk E) without their helper-side counterparts. Both sat as known-issues in the v0.2.0 release notes. v0.3 closes both loops: the PDF drop-zone now actually parses receipts via claude with the Read tool, and every filament — newly imported, manually added, or backfilled — gets a real Bambu hex (or, for non-Bambu, an AI-resolved best guess) instead of the grey `#888888` placeholder. Multicolor filaments come back with their full colour stops, and a new **Multicolor** filter on the Filaments page surfaces them as a first-class category.

A handful of v0.2 papercuts get fixed in the same release: the Save button on Add/Edit forms (Vue reactive Proxy vs `structuredClone`), the desktop launcher (four stacked Windows bugs masking a single IPv4/IPv6 mismatch), and a lenient JSON parser in the helper so claude's occasional preamble prose doesn't break structured-output endpoints.

This release was scaffolded by Chats A and C (Chat B's catalogue work landed earlier in the same `[Unreleased]` window).

## Highlights

### `/import-order` is real (Chunk F helper side)

`helper/index.mjs` gains `POST /import-order`. Pipeline:

1. Stream the multipart upload to a temp PDF under `os.tmpdir()` via `busboy`. 10 MB cap → `413 too_large`. Missing `pdf` field → `400 no_pdf_field`.
2. Spawn `claude --print --output-format json --model claude-sonnet-4-6 --allowedTools Read --permission-mode bypassPermissions --no-session-persistence --append-system-prompt <framing>` with `cwd=os.tmpdir()`. The cwd choice matters — see "Lessons" below.
3. The user-prompt asks claude to use the Read tool on the absolute temp path (Read handles PDFs natively, including vision-OCR for image-based scans) and return JSON matching the contract in `docs/parallel-work.md` §326–356.
4. Lenient parse, validate `items[]` is an array, force `ok: true`, return.
5. `finally` unlinks the temp PDF.

Verified end-to-end against a synthetic Bambu receipt PDF: `vendor_guess=bambu`, `order_ref`, `order_date` (ISO), 1 filament line item with `kind`, `brand`, `name`, `variant`, `sku`, `quantity`, `unit_price_eur`.

### Swatch resolver (Chunk E, Bambu-first)

`POST /api/lookup-swatch` replaces its v0.1 stub. Given `{ brand, name, variant?, sku?, color_code?, product_url? }`:

- **Bambu Lab** brand → dedicated prompt with the parsed colour code (e.g. `M8` from `A05-M8-1.75-1000-SPL`), WebFetches `eu.store.bambulab.com` / `wiki.bambulab.com` and the official Bambu hex-code PDFs. Source = `bambu`, confidence = `high` on exact SKU/code match, `medium` on variant-name match.
- **Other brands** → generic AI fallback. Source = `ai` / `generic`, confidence = `low`. 123-3d / RealFilament / SUNLU per-supplier resolvers are explicitly out of scope and stay on this path.

Result is cached in `data/ai-cache.json` under `swatch:<brand>|<name>|<variant>` so the same product never costs subscription quota twice. `force: true` bypasses the cache.

**Multicolor is first-class.** PLA Silk Multi-Color, PLA Galaxy, PLA Marble, Dual Color lines come back with every visible colour stop in `stops[]` and `effects` automatically includes `"multicolor"`. The "Ochtendglans" canary returned 4 stops — an orange/teal/purple/pink dawn gradient — matching the Dutch translation of the variant name ("dawn glow"). "Zuidstrand" returned 2 stops (pink + teal).

### Import-time swatch population (two-tier, never grey)

`/import-order` claude prompt is extended with `hex`, `stops[]`, and `effects[]` per filament line item — **Tier 1**, free (same call), best-guess from colour name + brand knowledge. The user sees a sensible colour the instant the review modal opens. `OrderImportReview.apply()` then fires `/api/lookup-swatch` per newly-created filament in parallel — **Tier 2**, post-import async, replaces the Tier 1 hint with the authoritative resolved swatch and persists via `filaments.save()`. If Tier 2 fails (helper offline, claude unauthenticated, timeout), the Tier 1 hint stays. **Drops never regress to grey** unless claude itself genuinely can't tell.

### Backfill — `scripts/backfill-swatches.mjs` + "Refresh swatches" button

Selection criteria: `swatch.hex==='#888888' && stops.length===1 && stops[0]==='#888888' && source==='manual'` (the legacy default for OrderImportReview-imported records before this release).

Two paths, one resolver:

- **CLI**: `node scripts/backfill-swatches.mjs`. Pre-flight `git status --porcelain data/filaments.json` refuses to run on a dirty working copy unless `SKIP_GIT_CHECK=1` is set. Atomic write at the end. Rollback = `git restore data/filaments.json`.
- **UI**: a fuchsia "Refresh swatches (N)" button (Palette icon) appears on the Filaments page next to "Fill missing AI" whenever grey-placeholder records exist. Progress inline, same persistence as manual saves.

Both share the resolver and the criteria — there's exactly one source of truth.

**Backfill executed on release:** all 13 grey filaments in `data/filaments.json` (Bambu TPU 85A/90A, PLA Matte refills, PLA Silk Multi-Color) replaced with resolved swatches sourced from the official Bambu hex-code PDFs. Two passes needed because the first run hit three WebFetch timeouts; second pass picked up two of three; the last one (PLA Matte refill "Appelgroen") failed on a JSON-with-prose preface and was caught by the `parseLooseJson` fix below on pass three.

### Multicolor filter on the Filaments page

New pseudo-family in the colour-family filter row, alongside Red/Orange/Yellow/etc. Match rule:

```ts
isMulticolor(f) =>
  f.swatch.effects.includes('multicolor')
  || (f.swatch.stops.length > 1 && new Set(f.swatch.stops).size > 1)
```

Swatch dot uses a CSS conic-gradient through all 12 single-colour family hues. The match logic short-circuits — a Galaxy filament shows up under Multicolor *only*, not also under its dominant colour bucket.

### Shopping list totals up (carried over from earlier in [Unreleased])

`ShoppingItem` gained `unit_price_eur?` snapshotted at pick time. The footer shows **Estimated total** for open items, separated from "+ €X already checked off" for done items. Free-text/consumable rows without a price render `—` and are reported as "N items without price not counted." The existing list was backfilled in-place on `ShoppingPage.vue` mount (idempotent, no-op when prices are already present).

### Catalog mirror of Bambu store P2S accessories (carried over)

`data/catalog/replacement-parts.json` grew from 33 → 62 items; `data/catalog/consumables.json` 16 → 18. Full AMS units (AMS 2 Pro `SAA034`, AMS HT `SAA036`), expansion hardware (Switching Adapter, 4-in-1 PTFE Adapter, Filament Buffer, TPU Feed Assist), exhaust kit, reusable spools, glues, the full Maker's Supply tool range, and the missing PTFE Tube 4000 mm `FAT003`. Every item carries Bambu SKU + EU EUR estimate + store URL + P2S-specific note.

## Fixed

- **Helper JSON parsing is now lenient.** New `parseLooseJson` helper: direct parse → strip-fences → extract first balanced `{...}` block. Caught a real failure during the swatch backfill (claude prefaced its JSON with "Found it confirmed in the raw GitHub colors.json…") and applies to `/lookup-filament`, `/lookup-swatch`, and `/import-order` uniformly.
- **PDF drop-zone "Failed to fetch"** — the v0.2 known issue (helper endpoint missing). Fixed by Chunk F.
- **Save button on Add/Edit forms now actually saves.** `FilamentForm.vue` and `AccessoryForm.vue` were calling `structuredClone(form)`; the form is a `reactive()` Vue Proxy whose nested objects (`swatch`, `inventory`) are also Proxies, which Chromium rejects with `DataCloneError`. The submit handler threw inside `@submit.prevent`, so the user just saw a dead button. Replaced with JSON-roundtrip clones.
- **Desktop launcher (`scripts/start.ps1`).** Four stacked Windows bugs — `npm.ps1` shim vs `npm.cmd`, missing Edge on PATH, UTF-8-without-BOM em-dash parse failure, and the user-visible root cause: `app/vite.config.ts` had `server.host: 'localhost'` resolving to `::1` first on Windows, leaving 127.0.0.1 unbound so `msedge --app=http://127.0.0.1:5173/` got connection-refused.

## Lessons / non-obvious

### `cwd: tmpdir()` when shelling out to `claude` from a helper

Spawning the user's local `claude` CLI inside the project root means claude auto-discovers `CLAUDE.md`, `crosslog.md`, and the project's auto-memory. For backend extraction endpoints (`/import-order`, `/lookup-swatch`) this is a problem: claude reads the surrounding helper/orchestrator context, decides the structured-JSON request looks scripted, and refuses as a suspected prompt injection. Reproduced with a fake-path test — claude returned a refusal message instead of JSON. Spawning from `os.tmpdir()` + adding `--append-system-prompt` framing the call as the legitimate extraction backend it is + `--no-session-persistence` keeps every call hermetic. Documented in `helper/README.md` so the next time someone touches this they don't reintroduce it.

### Two-tier swatch population beats a single-tier resolver

Doing only Tier 1 (extraction-time hex hint) yields wrong-but-plausible colours for unusual variants (Bambu's Dutch-named Silk Multi-Color lines). Doing only Tier 2 (post-import resolver) leaves the user staring at grey blobs for ~5 s while WebFetch chains complete. Combining both — Tier 1 instant, Tier 2 corrective and async — feels right: imports show colour immediately, then refine in the background. The fallback chain matters: Tier 2 failure must not regress Tier 1. Implemented by `OrderImportReview.resolveSwatchesAsync()`.

### `parseLooseJson` instead of `JSON.parse(stripFences(s))`

Claude with WebFetch sometimes prefaces structured output with a one-liner citation. `JSON.parse` is binary — it either works or doesn't. The lenient parser falls back to extracting the first balanced `{...}` block. Adopted across all `/lookup-*` and `/import-order` endpoints uniformly.

### Spec-before-execute survives session boundaries

Mid-session the user asked for "document and backfill first before executing so it can survive a session end". Wrote `docs/chunk-e-swatch-resolver.md` (full spec — endpoint contract, resolver strategy, backfill criteria, rollback, execution order, out-of-scope) before any code. The doc is the source of truth; if any code diverges, the doc loses. Made it possible to execute Chunk E across what would otherwise be a fragile multi-step plan.

## Known issues

- **Per-supplier swatch resolvers beyond Bambu remain on the AI-generic fallback.** 123-3d.nl / RealFilament / SUNLU stay on claude's training knowledge + optional WebFetch. Explicitly out of scope for v0.3 per `docs/chunk-e-swatch-resolver.md` §"Out of scope".
- **`/api/lookup-swatch` requires `claude` CLI auth.** Same as `/lookup-filament`. Failures surface inline; the resolver fails open (Tier 1 hint stays, "Refresh swatches" reports the count).
- **Backfill script refuses to run on dirty `data/filaments.json`** without `SKIP_GIT_CHECK=1`. By design.

## Deployment notes

No data migrations. `data/filaments.json` is touched by the in-release backfill (13 records now carry `source: 'ai'` swatches instead of `'manual'` greys) — diff is surgical. `data/ai-cache.json` gains `swatch:<brand>|<name>|<variant>` entries that the resolver populates on demand. New helper dep: `busboy@^1.6` already installed at `helper/node_modules/`. No frontend dependencies added.

For users already running the PWA: a single browser refresh picks up the new bundle. The live helper hot-swaps in place (the dev-server pattern from earlier sessions); standalone `node helper/index.mjs` users restart as usual.
