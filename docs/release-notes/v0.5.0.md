# Haspel v0.5.0 — 2026-05-24

## Summary

Haspel grows beyond a single printer. v0.5.0 adds **printer onboarding and
management** — register the machines you actually print on, see their full specs,
and let the catalog prefill the details — and broadens the whole app from the
Bambu Lab P2S to **any FDM printer**. It also folds in a documentation refresh for
the now-native (Tauri) app and a couple of fixes from v0.4.0 feedback.

## Highlights

- **Printers page.** Add, edit, remove, and set an active printer. Each printer
  has a detailed **spec view**: build volume, max build height, max hotend & bed
  temperature, enclosure / heated chamber, filament diameter, default + available
  nozzle sizes, AMS / multi-material unit, common accessories, and links to the
  brand/model page and brand store.
- **Prefill from a 65-model catalog.** The add form can prefill from a read-only
  seed (`data/catalog/printers.json`) covering **65 models across 11 brands**
  (Bambu Lab, Original Prusa, Creality, Anycubic, Elegoo, FlashForge, Sovol, QIDI,
  UltiMaker, Voron, AnkerMake). Specs were researched from official manufacturer
  sources, with `null` where a value couldn't be confirmed. Everything stays
  editable after prefill.
- **First-run prompt.** On a fresh install with no printers configured, opening
  the app offers to add your printer — **Add a printer / Not now / Don't ask
  again** (the last persists so it never nags again).
- **Any FDM printer.** The README, the in-app framing, and the user guide are
  broadened beyond the P2S; a new **Printers** chapter joins the user guide.

## Changed

- **Documentation refreshed for the native app.** The README, `docs/install.md`,
  and the user guide now describe installing and running the **Tauri app** (run
  the `.msi`/`.exe`, launch from the Start menu) rather than the old PowerShell
  `start.ps1` + dev-server flow — `start.ps1` is now documented as a dev-only
  convenience. `docs/architecture.md` gains the packaged-app (Tauri sidecar)
  process model.

## Fixed

- **`stest` no longer disturbs your real browser profile.** The fresh-user test
  harness used to open its window with a throwaway Edge `--user-data-dir` and
  force-kill Edge on teardown — which, when your normal Edge was running, fell
  back to the default profile and corrupted it. The test window now opens in the
  default profile (isolation already comes from the distinct `:5273` origin), and
  teardown stops only the helper + Vite, never the browser.
- **The first-run "Add a printer" button** now opens the form even when you're
  already on the Printers page (it now reacts to the route query instead of only
  firing on mount).

## Known issues

Carried over from v0.4.0 (unchanged):

- **One helper port.** The installed app and a dev/PWA instance both bind
  `127.0.0.1:5174` — don't run both at once.
- **Label PDF uses Edge headless** on the helper side; the cross-platform
  `window.print()` path is future work.
- **API-provider swatch** runs from model knowledge only (no live web); only the
  Claude CLI path fetches live colours.
- Per-supplier swatch resolvers beyond Bambu still use the generic AI fallback.

## Deployment notes

- No data migration needed. Printers are stored per-install in `printers.json`
  alongside your other data at `%APPDATA%\Haspel\data`.
- Install the `.msi` (per-machine) or `-setup.exe` (per-user, no admin) from
  `src-tauri/target/release/bundle/`. Data persists across reinstalls.
- Still not included (as in v0.4.0): code-signing (SmartScreen will warn),
  auto-updater, a CI release pipeline, and macOS/Linux builds.
